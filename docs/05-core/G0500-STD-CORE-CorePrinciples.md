# [STD-CORE] DocFoundaryフレームワーク中核原則（Core Principles）

# 1. 目的

本標準文書は、DocFoundary プロジェクトにおける文書整備および開発仕様管理の双方に共通する、最も上位に位置する基本原則（Core Principles）を定義する。

本標準は、個々の YAML 成果物（仕様、チェックリスト、指示書など）を詳細に定義するものではなく、それらの設計・利用・運用の指針として機能する抽象レイヤに位置付ける。

目的は以下の通り：

* 成果物の肥大化・冗長化を防ぐ
* 運用の儀式化（信仰遅延）を防止する
* 文書・仕様の整合性と更新性を確保する
* 文書と仕様の両領域を統一基盤で扱えるようにする

---

# 2. 適用範囲

本標準は以下のすべてに適用される：

* DocFoundary プロジェクトで扱うすべての成果物（Artifacts）
* 文書（Markdown, Word 等）
* YAML 成果物（Spec, Template, Checklist など種別は問わない）
* 設計資料、要求仕様、開発仕様、運用仕様
* それらの管理・運用・更新プロセス

ただし、本標準は具体的な形式定義（スキーマ）ではなく、上位の運用思想を定義するものであり、成果物の内容そのものは本標準に含めない。

---

# 3. 基本概念

成果物を扱うにあたり、以下の抽象概念を共通基盤とする。

| 概念        | 説明                                     |
| --------- | -------------------------------------- |
| Artifact  | 文書・YAML・コード等、管理対象となる成果物の抽象概念           |
| Metadata  | ID・バージョン・ステータス・関連などの付帯情報               |
| Structure | 文書・仕様・テンプレートの構造定義                      |
| Glossary  | 用語および定義。全成果物の表記統一の基盤                   |
| Relations | 成果物間の参照・依存・連携関係                        |
| Review    | レビュー項目・コメント・ステータス                      |
| Change    | 改訂履歴・変更要求・差分情報                         |
| Template  | 成果物構造の標準ひな形                            |
| Policy    | 文書/仕様/コーディングなどの規約                      |
| Lifecycle | Draft → Review → Approved → Deprecated |

これらはすべての成果物に共通するメタ層であり、個々の YAML 種別を定義するものではない。

---

# 4. 基本原則（Core Principles）

DocFoundary のすべての成果物は、以下の 10 原則に従う。

## 4.1 Single Source of Truth（唯一の真実の源）

* 全ての情報は一箇所のみで管理する。
* 複製・冗長表現は禁止し、ビュー側は参照に徹する。
* 仕様・文書・チェックリストの整合性を保つ基盤とする。

## 4.2 Minimalism（最小化の原則）

* 成果物は最小限に保ち、目的を満たす必要最低限だけ作成する。
* YAML種類の乱立を避け、作らない勇気を持つ。
* 必要が生じた時のみ、Just-In-Time で追加する。

## 4.3 Consistency（整合性の原則）

* 成果物は共通のメタデータ構造に従う。
* 表記揺れ・構造揺れを排除する。
* 参照関係は可能な限り明示的に記述する。

## 4.4 Abstraction（抽象化の原則）

* 文書と仕様は上位において同じ “Artifact” として扱う。
* 個別形式は下位概念であり、運用思想が優先される。
* 抽象概念により、運用の一貫性を維持する。

## 4.5 Traceability（関連性管理の原則）

* 成果物間の関係（満たす、依存する、参照する）を最小限で定義する。
* 必要十分なトレーサビリティのみを保持する。
* 過剰なリンク設定は避ける。

## 4.6 Separation of Concern（関心の分離）

* 文章（背景・理由）・構造・ルール・判断を混在させない。
* 文書は「物語」、仕様は「構造」として役割を分ける。
* 単一成果物の複雑化を防ぐ。

## 4.7 Composability（合成可能性の原則）

* 成果物は組み合わせ可能で、再利用可能な単位に分解する。
* モノリシックな巨大文書を避ける。
* 小さな意味単位を束ねて運用する。

## 4.8 Lifecycle Discipline（状態規律の原則）

* 成果物は最小限の状態遷移（Draft/Review/Approved/Deprecated）を持つ。
* 状態の曖昧化・放置を防ぎ、更新性を維持する。

## 4.9 Transparency（透明性の原則）

* 意思決定の理由（Rationale）を簡潔に記録する。
* なぜその成果物が必要なのかが分かるようにする。
* 形骸化した成果物の増殖を防止する。

## 4.10 Automation First（自動化優先の原則）

* 成果物は手作業ではなく自動化を前提とした形式（例：YAML）を推奨する。
* 文書生成・整合性チェック・レビュー補助の自動化を優先する。
* 人間への負荷と更新遅延を減らす。

---

# 5. 運用方針

本標準は成果物そのものを定義するものではなく、成果物を設計・作成・運用する際の上位理念として扱う。

* 個別の YAML スキーマや文書テンプレートは、本標準の下で必要に応じて策定する。
* 本標準に反する成果物の増殖を禁止する。
* 成果物の追加は、本標準の各原則に照らして慎重に判断する。

---

# 6. 今後の拡張

* 必要に応じ Core Principles を補強するための指針を追加する。
* YAML 個別仕様（Spec / Template / Checklist 等）は下位層として別文書で定義する。
* プロジェクト運用において新たな問題が生じた場合、本標準を優先して見直すこと。
